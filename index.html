Dependency Inversion Principle

High-level modules should not depend on low-level modules.
Both should depend on abstractions.

Was ist denn nun das Dependency Inversion Principle?
- semi-hilfreich ...
- mir hat Kontext gefehlt
- deswegen nicht weiter verwunderlich: erst verstanden, als ich mich mit etwas anderem befasst habe: Layer

Layers

Separate and organize code by it's role/responsibility in the system.
- Da ich das Dependency Inversion Principle anhand von Layern verstanden habe,
macht es für mich auch Sinn, es daran zu erklären.

Wofür Layer?
- allen Daten-Zugriffs-Code zusammen, allen für das Backend relevanten Frontend-Code zusammen, ...
- die Absicht des im Layer vorhandenen Codes explizit machen
- durch Layer austauschbar machen

- Wie bei Software-Design: es gibt nicht kein Design, sondern nur schlechtes Design
- Wie bei Software-Architektur: es gibt nicht keine Architektur, sondern nur Architektur, die man nicht unter Kontrolle hat
- So ist es auch bei Layern: es gibt nicht keine Layer, sondern nur solche, die man nicht berücksichtigt
  • Presentation
    Views, Controllers
  • Domain
    Entities, Domain Services, Domain Events
  • Persistence
    Repositories, ORM
- Ein Beispiel von Layern anhand von Web-App
- Request durchläuft
- Frontend-Layer: Welches Template; was wird gebraucht (Select-Station)
-- nutzt Domain-Layer: was wird angezeigt und wie?
- Domain-Layer: Business-Regeln; was darf angezeigt werden (ERV Enterprise SS)
-- nutzt Persistence-Layer: welche Daten zeige ich an?
- Persistence-Layer: Abstraktionsschicht für Datenzugriff (ERV Welche Versicherungen gibt es)
- letztendlich: Datenbank
- möchte 2 Layer im Detail sehen: Domain und Persistence

SwiftMailerAdapter


                        namespace Infrastructure;

                        use Domain\Mailer;

                        class SwiftMailerAdapter implements Mailer
                        {
                            public function __construct(\Swift_Mailer $mailer)
                            {
                                $this->mailer = $mailer;
                            }

                            public function send($recipient, $message)
                            {
                                $swiftMessage = new \Swift_Message();
                                $message->setTo($recipient);
                                $message->setBody($message);

                                $this->mailer->send($swiftMessage);
                            }
                        }
- wir brauchen das zu adaptierende Objekt
- dann brauchen wir das Interface, auf das wir adaptieren wollen; aus unserem Domain-Layer
- Interface gibt uns vor, was wir brauchen
- Hier und genau hier im SwiftMailerAdapter kommen die hässlichen Swift_Message-Details

- Nur verschoben? Schon, aber: Gartenschlauch in der Küche? Geschirr im Schlafzimmer?
-- Wenn man Dinge dort anordnet, wo man sie natürlicherweise erwarten würde, erleichtet man Leuten die Arbeit
-- Wie oft habt ihr schon Dinge gesucht und in den absurdesten Stellen gefunden?

Maintainability

Autonomy
Testability
Maintainability
"Separate code that changes from the code that stays the same!"
- Ergibt sich aus den beiden vorherigen
- je unabhängiger mein Code und je besser getestet, desto wartbarer wird er
- bspw. Unabhängigkeit: anderer Notification-Mechanismus -> keine Bugs in den Business-Regeln
- bspw. Testbarkeit: je mehr getestet mein Code ist, desto sicherer bin ich und desto weniger Bugs mache ich

- Business-Regeln wie "Schlechte Versicherung -> Schlechtes Angebot" wird sich nicht ändern
- Code, den ich nicht ändern muss, bleibt richtig
